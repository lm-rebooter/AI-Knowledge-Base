Architecture Map

AI Knowledge Base

这个项目把 UI、主业务后端、AI 能力和共享契约拆成了 4 个清晰层次。新人学习时最重要的不是一口气看完所有代码，而是先看清每一层只负责什么，再顺着“新增文档”或“聊天问答”两条主链路往下读。

Web: 页面与交互 API: 业务编排与数据库 AI Service: RAG 与 LLM Shared: DTO 与类型契约
apps/web

Next.js Web

负责所有用户可见页面、表单提交、聊天界面、知识库管理和工作台展示。

  • 页面路由: `app/`
  • 业务组件: `components/`
  • 请求封装: `lib/api.ts`
apps/api

NestJS API

负责登录认证、知识库 CRUD、文档解析与入库编排、聊天摘要统计以及数据库访问。

  • 主入口: `src/main.ts`
  • 根模块: `src/app.module.ts`
  • 文档模块: `src/modules/documents`
  • 聊天模块: `src/modules/chat`
apps/ai-service

FastAPI AI Service

负责文档切片、Embedding、向量检索、上下文拼装和最终的 RAG 问答生成。

  • HTTP 接口: `app/api/*`
  • RAG 主链路: `app/rag/*`
  • 向量存储: `app/vectorstore/*`
packages/shared

Shared Contracts

前后端共享 DTO、常量和响应类型，避免请求体和响应结构在多端漂移。

PostgreSQL

Structured Data

保存用户、知识库、文档正文、文档状态以及其他结构化业务数据。

Local Files / Vector Store

Runtime Storage

开发阶段保留本地上传文件、聊天 JSON 索引，以及 FAISS / Pinecone 相关向量数据。

Flow A: 新增文档

1. Web

用户在知识库页面上传 PDF 或粘贴文本。

2. API

NestJS 解析文件、清洗文本、写入 PostgreSQL。

3. AI Call

API 调用 FastAPI 的 `/api/ingest`。

4. RAG

AI Service 切片、Embedding、写入向量存储。

5. Result

文档变成可以在聊天里被检索命中的知识。

Flow B: 聊天问答

1. Web

用户在聊天页选择知识库并发起提问。

2. API

NestJS 保存会话上下文并转发问题。

3. Retrieve

AI Service 从向量存储召回最相关片段。

4. Generate

大模型结合检索上下文生成回答。

5. Return

答案回到 Web，并同步更新本地会话与统计。

建议新人的第一条阅读路径是 “新增文档”，第二条再看 “聊天问答”。这样能最快理解项目的真正主链。
